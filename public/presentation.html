핵심 컨셉

"감정이 글씨체가 된다"

일반 메모 앱은 글자 모양이 늘 똑같습니다. typing.dot은 문단을 쓸 때마다 그 텍스트의 감정을 분류하고(기쁨·즐거움·평온·슬픔·우울·불안·분노·중립), 감정마다 묶여 있는 손글씨 폰트 풀에서 하나를 골라 적용합니다. 같은 감정이라도 블록마다 폰트를 무작위로 바꿔, 손으로 쓴 듯한 리듬이 생깁니다.

단발성 분류가 아니라 감정 모멘텀 규칙을 둡니다. 분류 실패 시 직전 감정을 유지하고, '중립'은 최근 감정과 블렌딩하며, 같은 감정이 연속되면 확정합니다. 덕분에 글의 흐름이 급격히 튀지 않고 자연스럽게 이어집니다.

역할 ① 배포 인프라

▲ Vercel이 하는 일

정적 빌드 & 글로벌 CDN

vite build 산출물(HTML·JS·CSS)을 전 세계 엣지에 캐싱·서빙. 서버 런타임 없이 빠른 로딩.

Git 연동 자동 배포

main 브랜치 푸시 → 자동 빌드·배포. 커밋마다 고유 프리뷰 URL, 프로덕션은 typing-dot.vercel.app 별칭.

SPA 라우팅

vercel.json의 rewrite로 모든 경로를 /index.html로 보내, 클라이언트 라우터(/login, /friends 등)가 처리.

환경변수 주입

빌드 시 VITE_SUPABASE_URL · VITE_SUPABASE_ANON_KEY를 번들에 주입. 공개돼도 안전한 값만 노출.

※ VITE_ 접두 변수는 빌드 타임에 코드에 박힙니다 → 값을 바꾸면 반드시 재배포해야 반영됩니다.

역할 ② 백엔드 전체

Supabase가 하는 일

하나의 관리형(BaaS) 플랫폼으로 인증·DB·실시간·서버리스 함수를 모두 담당합니다.

기능Supabase 구성요소이 프로젝트에서의 쓰임
인증 Auth (GoTrue) 이메일·비밀번호 회원가입/로그인, 이메일 인증, 세션 토큰(JWT) 자동 갱신. 가입 시 트리거로 프로필 + 개인 일기장 자동 생성.
데이터 Postgres + RLS profiles, diaries, diary_members, blocks, friendships, diary_invites, subscriptions 7개 테이블. 행 단위 접근제어(RLS)로 "내 데이터만" 보장.
실시간 Realtime blocks 테이블 변경을 WebSocket으로 구독 → 공유 일기장에서 다른 멤버의 글이 즉시 반영.
서버 로직 Edge Functions (Deno) analyze-emotion(AI 프록시 + 쿼터 강제), delete-account(service role로 계정·데이터 완전 삭제).
저장 프로시저 SQL RPC username_available, accept_diary_invite, my_diary_invites, diary_members_list 등 권한이 필요한 작업을 안전하게 캡슐화.

왜 RLS가 핵심인가

브라우저가 DB에 직접 쿼리하는 구조라, 보안을 "숨김"이 아니라 "정책"으로 강제합니다. 예: blocks는 is_diary_member(diary_id)가 참인 사용자만 읽기/쓰기 가능 — 남의 일기 ID를 알아도 정책에서 막힙니다. anon key가 공개돼도 안전한 이유입니다.

데이터 모델의 한 장면

캔버스의 글 한 덩어리 = blocks 한 행. 위치(x,y), 텍스트, 감정, 폰트, 그리고 글자별 스타일(char_styles jsonb)과 타이핑 리듬(strokes)까지 담아, 화면의 모습이 손실 없이 저장·복원됩니다. 각 행은 작성자(author_id)와 일기장(diary_id)에 묶이고 RLS가 접근을 통제합니다.

동작 흐름

"한 문장을 쓰면 벌어지는 일"

타이핑 — 캔버스 블록에 글 입력. 글자별 스타일이 charStyles로 즉시 기록.
임계값 게이트(클라이언트) — 15자/3단어 미만이면 AI 호출 없이 스킵해 쿼터 절약.
Edge Function 호출 — supabase-js가 세션 JWT를 실어 analyze-emotion 호출.
서버 검증·분류 — JWT로 사용자 확인 → 티어/쿼터 점검 → OpenRouter에 분류 요청 → 사용량 +1.
감정 모멘텀 반영 — 받은 감정을 최근 이력과 합쳐 최종 감정·폰트 결정, 글씨체 전환.
자동저장 — 700ms 디바운스로 blocks upsert. 공유 일기장이면 Realtime으로 다른 멤버 화면에도 반영.

용어 사전

발표에 나온 용어 풀이

처음 듣는 용어가 있어도 괜찮습니다. 발표에서 쓰는 기술 용어를 쉬운 말로 정리했습니다.

웹 · 배포

SPA Single Page Application
페이지를 통째로 새로 불러오지 않고, JavaScript가 화면 일부만 바꿔주는 웹앱 방식. 전환이 빠릅니다.
Jamstack
미리 빌드된 정적 프런트엔드 + API 백엔드로 역할을 나누는 구조. 빠르고 보안·확장이 쉽습니다.
CDN Content Delivery Network
파일을 전 세계 여러 서버에 복제해 두고, 사용자와 가장 가까운 곳에서 전달하는 망.
빌드 / 빌드 타임 Build
소스코드를 브라우저가 읽을 수 있는 파일로 변환하는 과정(여기선 Vite). "빌드 타임에 박힌다"는 그 변환 시점에 값이 코드에 고정된다는 뜻.
환경변수 Env Variable
코드 밖에서 주입하는 설정값(접속 주소, 키 등). 개발/운영 환경마다 다르게 줄 수 있습니다.
rewrite
들어온 주소를 서버가 다른 경로로 바꿔 처리하는 규칙. SPA에선 모든 경로를 index.html로 보냅니다.

백엔드 · 데이터

BaaS Backend as a Service
인증·DB·서버로직을 직접 구축하지 않고 빌려 쓰는 관리형 백엔드(여기선 Supabase).
서버리스 / Edge Function Serverless
서버를 직접 운영하지 않고 "함수 단위"로 실행되는 코드. Edge는 사용자와 가까운 위치에서 실행됩니다.
Postgres
널리 쓰이는 오픈소스 관계형 데이터베이스(표 기반).
jsonb
Postgres에서 JSON 데이터를 효율적으로 저장하는 컬럼 타입.
upsert
insert(삽입) + update(수정). 행이 없으면 새로 넣고, 있으면 덮어씁니다.
RPC Remote Procedure Call
DB 안에 미리 정의해 둔 함수를 클라이언트가 원격으로 호출하는 것.
트리거 Trigger
특정 DB 이벤트(예: 회원가입) 발생 시 자동 실행되는 함수.
마이그레이션 Migration
① DB 스키마 변경을 버전으로 관리하는 SQL 파일 ② 로컬 데이터를 클라우드로 옮기는 작업. 이 프로젝트는 둘 다 사용합니다.

보안 · 인증

Auth Authentication
"당신이 누구인지" 확인하는 과정(로그인).
JWT JSON Web Token
로그인 후 발급되는 서명된 토큰. 매 요청에 실어 신원을 증명하며, 위조하면 서명 검증에서 걸립니다.
RLS Row Level Security
데이터베이스가 행마다 "이 사용자가 접근해도 되는지"를 정책으로 검사. 백엔드 코드 없이 권한을 강제합니다.
anon key
브라우저에 공개되는 익명 키. 이 키만으론 RLS를 못 넘으므로 노출돼도 안전합니다.
service role key
모든 권한을 가진 관리자 키. 절대 브라우저에 두지 않고 서버(Edge Function)에만 둡니다.
실시간 / WebSocket
서버가 변경 사항을 실시간으로 밀어주는 양방향 지속 연결. 새로고침 없이 화면이 갱신됩니다.

AI

LLM Large Language Model
방대한 텍스트로 학습한 대규모 언어모델. 여기선 글의 감정을 분류합니다.
OpenRouter
여러 LLM 제공사를 하나의 API로 중계해 주는 서비스.
gpt-4o-mini
빠르고 저렴한 경량 모델. 짧은 감정 분류 작업에 적합합니다.
토큰 Token
모델이 텍스트를 처리하는 단위(대략 단어 조각). 응답 길이를 토큰 수로 제한합니다.
쿼터 Quota
사용 한도. 이 앱은 무료 등급 AI 분석을 하루 20회로 제한합니다.

프런트엔드 · 기타

디바운스 Debounce
연속으로 일어나는 이벤트(타이핑)를 모아 마지막에 한 번만 처리. 불필요한 저장 호출을 줄입니다.
ErrorBoundary
React에서 하위 컴포넌트의 오류를 잡아, 앱 전체가 멈추지 않도록 격리하는 장치.
무한 캔버스 Infinite Canvas
경계 없이 자유롭게 이동(패닝)·확대/축소(줌)하는 화면.
훅 Hook
React에서 상태·로직을 함수로 묶어 재사용하는 방식(예: useCanvasData).

QA 대비

anon key가 번들에 노출되는데 위험하지 않나요?
안전합니다. anon key는 설계상 공개되는 publishable 키입니다. 실제 권한은 키가 아니라 로그인으로 발급된 JWT와 Postgres RLS 정책이 결정합니다. 즉 키를 알아도 RLS를 통과하지 못하면 데이터를 읽거나 쓸 수 없습니다. 반대로 service role 키처럼 위험한 키는 Edge Function 서버 환경변수에만 둡니다.
AI 키(OpenRouter)는 어떻게 보호하나요? 사용량 한도는 우회 가능한가요?
AI 키는 브라우저에 전혀 내려가지 않고 analyze-emotion Edge Function의 서버 시크릿으로만 존재합니다. 클라이언트는 그 함수를 호출할 뿐 키를 볼 수 없습니다. 일일 한도(free 20회)도 서버에서 프로필의 사용량을 확인해 차단하므로 클라이언트 코드를 조작해도 우회되지 않습니다.
공유 일기장 실시간 동기화는 어떻게 동작하나요?
Supabase Realtime이 Postgres의 blocks 변경(insert/update/delete)을 WebSocket으로 푸시합니다. 클라이언트는 diary_id로 필터링한 채널을 구독하고, 자기 자신이 쓴 변경(에코)은 무시해 중복을 막습니다. 다른 멤버의 블록만 화면에 즉시 추가/갱신됩니다.
로그인 안 하면 못 쓰나요? 오프라인은요?
로그인 없이도 localStorage 기반 로컬 일기로 동작합니다(감정 분석만 로그인 필요). 나중에 로그인하면 마이그레이션 배너가 떠 로컬 글을 클라우드로 옮길 수 있습니다. Supabase 설정이 없는 환경에서도 앱이 죽지 않고 로컬 전용으로 폴백하도록 클라이언트를 가드했습니다.
왜 직접 서버를 만들지 않고 Vercel + Supabase인가요?
운영 부담을 줄이고 핵심 기능에 집중하기 위해서입니다. 프런트엔드는 Vercel이 CDN으로 서빙하고, 인증·DB·실시간·서버리스 함수는 Supabase가 관리형으로 제공합니다. 서버 프로비저닝·스케일링·보안 패치를 직접 하지 않아도 되고, RLS로 백엔드 코드 없이도 안전한 데이터 접근을 구현할 수 있습니다.
감정 분석 정확도가 낮거나 실패하면 화면이 이상해지지 않나요?
실패·모호한 경우 unclassified로 처리하고 직전의 의미 있는 감정을 유지합니다. 모멘텀 규칙이 급격한 폰트 변화를 막아, 한 번의 오분류로 글씨체가 튀지 않습니다. 네트워크/쿼터 문제도 사용자에게 안내 메시지를 보여주고 이전 상태를 보존합니다.
회원 탈퇴 시 데이터는 정말 삭제되나요?
네. delete-account Edge Function이 service role 권한으로 auth 사용자와 연결 데이터를 삭제합니다. 스키마의 외래키가 on delete cascade라 프로필·일기·블록 등이 함께 정리됩니다.
